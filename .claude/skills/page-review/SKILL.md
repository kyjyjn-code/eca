# 화면 통합 점검 (/page-review)

세 에이전트를 **병렬로** 띄워 한 보고서로 합친다. 합치는 것은 **보고서**이지 에이전트가 아니다 — 각자 자기 정본만 읽어야 정확도가 유지된다.

> **세 축(web-design-auditor · ux-reviewer · interaction-flow-architect)은 전부 돈다. 하나라도 빼면 이 스킬이 아니다.** 손으로 일부 에이전트만 재현하지 말고, 이 스킬을 통째로 실행한다. 특히 시각 디자인 축(web-design-auditor)이 조용히 누락되기 쉬우니, 실행 전 "세 축을 다 띄웠는가"를 확인한다. 화면이 많아 여러 배치로 나눠 돌리더라도 **각 배치가 세 축을 모두 포함**해야 한다.

## 0. 먼저 확인 — 만들기와 점검이 섞였는가

같은 턴에 파일을 만들거나 고친 직후라면 **점검을 거부하고 알린다.** 만든 사람이 자기 걸 검사하면 의미가 없다.

> "방금 수정한 파일입니다. 점검은 다음 턴에 별도로 요청해 주십시오."

사용자가 그래도 하라고 하면 진행하되, 보고서 머리에 **"작성 직후 자가 점검 — 놓친 것이 있을 수 있음"**을 명시한다.

## 1. 입력 확정 (되묻기는 한 번에 몰아서)

세 에이전트가 각각 물어보게 두지 않는다. **여기서 한 번에** 확정한다.

| 항목 | 쓰는 곳 | 없으면 |
|---|---|---|
| **대상 파일 경로** | 셋 다 | Glob으로 후보 제시 후 확인 |
| **사용자층** (학부모/학생/교사) | ux-reviewer, interaction-flow-architect | 파일 맥락에서 판단, 애매하면 묻는다 |
| **핵심 과업** (사용자가 끝내야 할 일) | interaction-flow-architect | 파일 맥락에서 판단, 애매하면 묻는다 |
| **SDC 프로젝트 여부** | web-design-auditor | 경로가 `2. SDC/` 아래면 자동 판정 |

- 파일에서 읽어낼 수 있으면 **읽어서 판단하고, 판단 근거를 한 줄로 밝힌 뒤 진행한다.** 확인만 받으려고 멈추지 않는다.
- 정말 갈리는 경우(학부모용인지 교사용인지에 따라 폼 구조 판정이 반대로 뒤집힘)에만 **묻고 멈춘다.**
- 대상이 여러 파일이면 그대로 전부 넘긴다. 파일별로 나눠 보고한다.

## 2. 병렬 실행

세 에이전트를 **한 메시지에서 동시에** 띄운다. 순차로 돌리지 않는다.

| 에이전트 | 축 | 권한 |
|---|---|---|
| `web-design-auditor` | 배치·위계·시각 브랜드 | **읽기 전용** |
| `ux-reviewer` | 한 화면 내부의 사용성·접근성 | 승인 후 수정 가능 |
| `interaction-flow-architect` | 화면 사이의 흐름·상태·전환 | 승인 후 수정 가능 |

각 에이전트에 **1단계에서 확정한 정보를 그대로 전달**한다. 다시 묻지 않도록 프롬프트에 명시한다.

> 예: `report.html을 점검하라. 사용자층=학부모(모바일), 핵심 과업="성적 확인 후 상담 신청", SDC 프로젝트임. 이 정보로 충분하니 되묻지 말고 진행하라.`

## 3. 병합 규칙

**중복 제거 — 같은 줄, 같은 원인이면 하나로 합친다.**
경계를 겹치지 않게 설계했지만 실제로는 겹칠 수 있다. 판정 기준:

| 겹치는 항목 | 채택할 쪽 | 이유 |
|---|---|---|
| 터치 타깃 크기 | `ux-reviewer` | 접근성 기준선이 정본 |
| 색만으로 상태 표현 | `ux-reviewer` | 접근성 문제 |
| 색 대비 부족 | `ux-reviewer` | WCAG 조항 |
| 버튼 개수·강조 과다 | `web-design-auditor` | §4.5.1 위계 상한 |
| 모달 사용 여부 | `interaction-flow-architect` | 흐름 판정 |
| 빈 상태·오류 상태 유무 | `interaction-flow-architect` | 상태 설계 |

같은 지적이 두 곳에서 오면 **채택한 쪽의 조항으로 한 번만** 싣고, 다른 쪽이 같은 것을 봤다는 사실은 적지 않는다(노이즈).

**상충하면 숨기지 않는다.** 두 에이전트의 권고가 반대 방향이면 양쪽을 다 싣고, 어느 쪽이 이 화면에 맞는지 근거와 함께 판단해 제시한다.

## 4. 출력 형식 (한국어)

심각도 순으로 정렬한다. **영역 태그를 반드시 단다.**

```
# 화면 점검 — <파일명>
대상: 학부모(모바일) · 핵심 과업: 성적 확인 후 상담 신청 · SDC

## 심각도 4·3 (먼저 고쳐야 함)
1. [흐름/§8-4] index.html:142 `<select onchange="location.href">` — 값 고르는 순간 페이지 이동
   근거: 컨트롤 설정 변경만으로 맥락이 바뀌면 안 됨 [WCAG 3.2.2 Level A]
2. [사용성/§3-1] index.html:88 이메일 입력에 <label> 없음, placeholder만
3. [흐름/§3-1] report.html:210 성적 목록에 빈 상태 처리 없음

## 심각도 2·1 / 주의
1. [배치/§4.5.3] report.html:240 카드 padding 24px > 컨테이너 gap 16px — 내부가 외부보다 넓어 그룹이 안 묶임
2. [배치/§4.5.1] font-size 값 7종 — 상한 3단계 초과

## 확인 요청 (사람 판단·렌더링 필요)
1. [배치/§4.5.5] 히어로 좌우 균형 — 코드로 판정 불가
2. [흐름/§4-2] 삭제 확인 모달 3곳 — 되돌릴 수 없는 액션인지 확인 필요

## 통과
배치: 제목-본문 비율 2.1배 / 주 버튼 1개 / 스페이싱 스케일 준수
사용성: focus 관리 / word-break:keep-all / inputmode
흐름: 뒤로 링크 / Escape 닫기 / 포커스 복귀

## 수정 순서 제안
1차 [흐름] 4·3건 — 화면·상태가 추가되므로 먼저
2차 [사용성] N건
3차 [배치] N건 — 콘텐츠 확정 후 마지막
```

- 지적마다 **영역 태그 + 파일:줄번호 + 조항(§) + 출처** 넷을 단다. 하나라도 없으면 싣지 않는다.
- 통과 항목은 축별로 나열만 한다. 설명하지 않는다.
- 마지막에 "어느 것부터 적용할까요?" 한 줄.

## 5. 수정 적용

**수정 순서는 흐름 → 사용성 → 배치다.** 흐름 수정은 화면·상태를 추가하는 경우가 많아(빈 상태 화면 신설 등) 나중에 하면 배치 작업을 다시 하게 된다.

- `web-design-auditor` 지적은 **그 에이전트가 고치지 않는다**(읽기 전용). 승인받으면 내가 직접 고치거나 해당 담당에게 넘긴다.
- 나머지 둘은 각자 **미리보기(목업 또는 As-Is/To-Be 다이어그램)를 제시한 뒤** 승인을 받아 고친다. 이 단계를 건너뛰지 않는다.
- **승인한 항목만** 고친다.

## 금지사항

- **세 축 중 하나라도 빼고 진행 — 금지.** web-design-auditor(시각 디자인) 누락이 대표 실수다. ux·flow 두 축만 돌리고 "점검 끝"이라 보고하지 않는다.
- 세 에이전트를 순차 실행 — 금지. 병렬로 띄운다.
- 세 에이전트에 정본을 대신 읽어 요약해 넘기기 — 금지. 각자 자기 정본을 읽어야 조항 번호가 정확하다.
- 같은 지적을 영역만 바꿔 두 번 싣기 — 금지.
- 상충하는 권고를 한쪽만 싣고 조용히 버리기 — 금지.
- 사용자층·핵심 과업을 확정하지 않은 채 실행 — 금지. 에이전트들이 각자 되물어 대화가 세 갈래로 갈라진다.
- 만든 직후 같은 턴에 점검 — 원칙적으로 금지(§0).
